WHAT THIS APP DOES — COMPLETE FUNCTIONALITY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WHO USES IT:
  VEGA professionals — cleaners, car washers, etc.
  Workers are SALARIED EMPLOYEES, not gig workers.

IMPORTANT RULE — HARDCODED IN APP:
  🚫 NEVER shows job amount / price / earnings to worker
  🚫 Workers do NOT know what customer paid

──────────────────────────────────
1. AUTHENTICATION
──────────────────────────────────

✅ DONE — Phone OTP via Firebase Auth
  - Worker enters their registered mobile number
  - Real SMS OTP sent by Firebase (requires Blaze plan)
  - On verification:
      → Checks Firestore workers collection
      → If not found: "Access Denied"
      → If status = "suspended": "Account Suspended. Contact hub manager."
      → If status = "blocked": "Account Blocked. Contact admin."
      → If status = "active": Login success

Firestore check:
  collection: workers
  query: where phone == entered_number
  fields checked: status, isAvailable, name, services

FCM token registered on every login for push notifications.

──────────────────────────────────
2. JOBS TABS
──────────────────────────────────

✅ DONE — 4 job tabs with real-time Firestore listener:

  📅 TODAY TAB
    → Shows all jobs assigned to this worker for today
    → Filters by createdAt date matching today
    → Badge count on tab icon (unread/pending jobs)

  ⚡ ACTIVE TAB
    → Shows jobs with status: assigned, on_the_way, in_progress
    → Badge count shows urgent attention needed
    → Sorted by createdAt desc

  ✅ COMPLETED TAB
    → Shows all completed jobs
    → No pricing shown — just service + customer name + time

  📋 ALL JOBS TAB
    → Complete history of all assigned jobs
    → All statuses visible

Each job card shows:
  - Order ID (last 6 chars)
  - Customer name
  - Service type (Home Cleaning, Car Wash, etc.)
  - Address (first 50 chars)
  - Scheduled time/slot
  - Status badge (colored)
  - Photo upload status (Before: X, After: X)
  - Delay flag warning if applicable
  🚫 NO job amount or price

──────────────────────────────────
3. JOB FLOW — STEP BY STEP
──────────────────────────────────

✅ DONE — Complete job lifecycle:

STEP 1: ASSIGNED
  Worker sees job in Today/Active tab
  Quick action button: "🚀 Start Going"
  → Updates Firestore:
      status = "on_the_way"
      onTheWayAt = server timestamp
      delayToStart = minutes since assignedAt
      delayFlag = true if delay > 15 minutes

STEP 2: ON THE WAY
  Worker taps job card → opens Job Detail screen
  Shows customer address + navigation button
  Shows "Enter Customer OTP" section

  BEFORE entering OTP:
    → Worker MUST upload at least 1 BEFORE photo
    → If no before photos: "Upload BEFORE photos first"
    → Upload button opens photo simulation
      (In production: opens camera/gallery)

  Worker enters 4-digit OTP from customer
  → Verifies against Firestore job.otp field
  → Wrong OTP: "Ask customer again"
  → Correct OTP:
      status = "in_progress"
      startedAt = server timestamp
      otpVerified = true
      delayToArrive = minutes since onTheWayAt
      delayFlag = true if delay > 15 minutes

STEP 3: IN PROGRESS
  Worker performs service
  Quick action button: "🎉 Mark Job Complete"

  BEFORE completing:
    → Worker MUST upload at least 1 AFTER photo
    → If no after photos: "Upload AFTER photos first"

  On completion:
    → status = "completed"
    → completedAt = server timestamp
    → workers/{id} updated:
        totalJobsCompleted += 1
        attendance.jobsToday += 1
        attendance.jobsWeek += 1
    🚫 NO earnings/payment fields updated

──────────────────────────────────
5. JOB REJECTION
──────────────────────────────────

✅ DONE — Worker can reject with reasons

Available reasons:
  - Too far from my location
  - Not available right now
  - Health issue
  - Safety concern
  - Other (free text)

On rejection:
  → Firestore update:
      status = "rejected"
      rejectedBy = workerId
      rejectedByName = worker name
      rejectReason = selected reason text
      rejectedAt = server timestamp
      assignedWorkerId = null (unassigned)
  → Hub Manager notified (via FCM — if server-side setup)

Can reject when status is: assigned, on_the_way

──────────────────────────────────
6. DELAY TRACKING
──────────────────────────────────

✅ DONE — Automatic delay detection

Tracked timestamps:
  - assignedAt (set by hub manager when assigned)
  - onTheWayAt (set when worker taps "Start Going")
  - startedAt (set when OTP verified)
  - completedAt (set when job marked complete)

Delay calculations:
  delayToStart = minutes from assignedAt to onTheWayAt
  delayToArrive = minutes from onTheWayAt to startedAt

If any delay > 15 minutes:
  → delayFlag = true saved in Firestore
  → "⚠️ Delayed" badge shown on job card
  → Hub manager sees this flag

FIRESTORE COLLECTIONS USED
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

READS FROM:
  workers/{phone} → worker profile, status, availability
  jobs where assignedWorkerId == workerId → my jobs

WRITES TO:
  jobs/{jobId}:
    status, onTheWayAt, startedAt, completedAt
    otpVerified, beforePhotos, afterPhotos
    delayFlag, delayToStart, delayToArrive
    rejectedBy, rejectReason, rejectedAt

  workers/{id}:
    isAvailable, fcmToken, lastLocationAt
    totalJobsCompleted (increment)
    attendance.jobsToday (increment)
    attendance.jobsWeek (increment)

NEVER READS:
  jobs/{jobId}.amount
  jobs/{jobId}.total
  jobs/{jobId}.price
  Any pricing or revenue field

HOW TO TEST THE APP
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

TEST 1 — Login
  1. Install APK on Android phone
  2. Open app → enter your phone number
  3. Wait for SMS OTP
  4. Enter OTP
  Expected: If number is in workers collection → login success
  Expected: If number not in workers → "Access Denied"

TEST 2 — Blocked Worker Login
  1. In Firebase Console, set a worker's status = "suspended"
  2. Try to login with that number
  Expected: "Account Suspended" error — cannot login

TEST 3 — Job Flow (end-to-end)
  Prerequisites:
    → Customer places a booking (via Customer App)
    → Hub Manager assigns that booking to this worker
  
  1. Login as worker
  2. Go to Today tab — see the assigned job
  3. Tap "Start Going" → see status change to "On the Way"
  4. Check Firebase Console → job.status = "on_the_way" ✅
  5. Tap job → tap "Before Photos" → add photo
  6. Enter 4-digit OTP (from customer's app)
  7. If OTP correct → job starts
  8. Check Firebase Console → job.status = "in_progress" ✅
  9. Tap "After Photos" → add photo
  10. Tap "Mark Job Complete"
  11. Check Firebase Console → job.status = "completed" ✅
  12. Check workers collection → totalJobsCompleted increased ✅

TEST 4 — Without Before Photos
  1. Tap job → reach OTP screen without uploading photos
  2. Enter correct OTP
  Expected: "Upload BEFORE photos first" alert — cannot start

TEST 5 — Without After Photos
  1. Job is in_progress
  2. Tap "Mark Complete" without after photos
  Expected: "Upload AFTER photos first" alert — cannot complete

TEST 6 — Reject Job
  1. Open assigned job
  2. Scroll to bottom → "Reject This Job"
  3. Select reason → Confirm
  Expected: Job status = "rejected" in Firebase Console ✅
  Expected: Job is removed from worker's list ✅

TEST 7 — Delay Flag
  1. Assign a job to worker
  2. Wait 20+ minutes before tapping "Start Going"
  Expected: delayFlag = true in Firebase Console ✅
  Expected: "⚠️ Delayed" badge on job card ✅

TEST 8 — Work Summary
  1. Complete a job
  2. Go to Summary tab
  Expected: jobsCompletedToday increases by 1 ✅
  Expected: NO price or ₹ amounts visible anywhere ✅

TEST 9 — Availability Toggle
  1. Profile tab → toggle availability OFF
  Expected: workers/{id}.isAvailable = false in Firebase ✅
  Expected: Hub Manager sees this worker as "On Job/Unavailable" ✅
